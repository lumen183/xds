## 测试环境

### 测试仓库
基于github仓库 mycastiel/xds 开展。当前实际跑通并取得 benchmark 数据的环境是 Linux 4.19.90、单 NVMe、0 号 NPU 卡。

内网环境的 xds 库有更多适配功能，比如对linux 6.6.0的支持，多种存储设备的支持，多卡支持等等，由于其内部测试还未完成，本次测试只使用了 mycastiel/xds 仓库的功能。

### 测试设备

4.19.90的机子 10.170.27.89 并没有实现多raid，实测仅有设备`/dev/nvme0n1`，挂载到了`workspace`，其他设备没有具体挂载点。因此在测试中仅使用了单卡单设备的场景。

因此本次测试中的最大理论性能为 7.34 GiB/s。


## benchmark 测试结果

### 测试局限

当前的benchmark没有重复测试，因此所有数据应该有+- 0.3GiB/s 左右的误差。

同时，benchmark多次测试的情况下会引发整机重启，可能是驱动或者硬件问题，暂时未知。但也说明当前脚本在连续压力测试下可能存在稳定性问题。

### 测试数据
具体的性能曲线如html中所示。

这里给出几个关键点的性能数据：
1、在大粒度大io-depth的情况下，可以实现 6.30 GiB/s 的最大性能，接近理论上限 7.34 GiB/s。
2、在小粒度读取情况下，软件开销是主要性能瓶颈。

从Python端和CPU端在32KB，64KB的小粒度读取下的性能数据来看，基本是翻倍关系，说明软件开销是主要瓶颈。

同时，CPU 端性能在 Python 端达到 6.0GB/s 左右的最大性能时，相较 Python 端也没有多大的提升。

## 后续建议

1、目前的github仓库还有部分bug，比如nvme调用错误，仍然返回0，导致drain_read()返回0，无法正确判断是否读取完成。建议在正式验收前修复这些bug。

还有只支持npu:0，不支持较新的 linux 内核等问题，需要和os部门沟通，尽快在内网环境中完成对 xds 库的适配和测试。

2、在内网环境中完成对 xds 库的适配和测试后，建议在多盘raid、多卡并发和非 0 卡读取等场景下进行测试，验证其性能和稳定性。

3、目前的benchmark还有一定局限，如稳定性，重复测试等，后续需要完善。
