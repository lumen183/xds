01 · EXECUTIVE SUMMARY

执行摘要

本次工作基于公开仓库 mycastiel/xds 开展，不以能力更完整的内网版本为基础。当前实际跑通并取得 benchmark 数据的环境是 Linux 4.19.90、单 NVMe、0 号 NPU 卡。

内网环境的 xds 库有更多适配功能，比如对linux 6.6.0的支持，多种存储设备的支持，多卡支持等等，由于其内部测试还未完成，本次测试只使用了 mycastiel/xds 仓库的功能。 另外，4.19.90的机子 10.170.27.89 并没有实现多raid，实测仅有设备`/dev/nvme0n1`，挂载到了`workspace`，其他设备没有具体挂载点。因此在测试中仅使用了单卡单设备的场景。

02 · SCOPE & LIMITS

版本范围与能力边界

本次评估对象

  • 基于 GitHub 公开仓库 mycastiel/xds 开发。
  • 以公开仓库 origin/main 的实现作为问题分析基线。
  • 本地分支中虽有 Linux 6.6 兼容性修改，但仍属于适配与验证过程。
  • 当前 benchmark 的实测运行内核为 Linux 4.19.90。

与内网版本的差异

  • 内网版本对 Linux 6.6.0 有更成熟的适配。
  • 内网版本对多种存储设备有更多支持。
  • 上述能力不能直接视为公开仓库版本已经具备。
  • 公开版本当前应按 4.19.90、有限设备组合进行支持声明。

128 KiB 限制应如何理解

origin/main 中 HW_LIMIT_SIZE = 128U << 10 限制的是单条底层 NVMe 命令，不是上层 P2P 请求的绝对最大值。256K、512K 等请求可以提交，但会在驱动内拆成多条不超过 128 KiB 的命令。

影响

大粒度请求可以运行并取得较高吞吐，但拆分会增加命令数量、调度成本和页边界处理复杂度。因此报告不将其表述为“超过 128K 完全不能读”，而是“单条底层 I/O 上限为 128 KiB”。

设备与多卡边界

接口会传递 devid/vfid，但参数存在不等于功能已经验证。公开版本缺少多卡并发编排、拓扑检查、聚合吞吐统计和完整真机回归；本测试环境中非 0 卡读取未跑通，因此当前阶段按仅支持 0 号卡处理。

/proc/mdstat 为空只能说明没有检测到 Linux MD 软件 RAID，不能单独排除硬件 RAID、设备侧 RAID 或其他卷管理方案。本报告按“当前无 MD RAID、单 NVMe 测试”分析。

7.34 GiB/s 的来源

PCIe 4.0 单 lane 速率为 16 GT/s，采用 128b/130b 编码。x4 链路的编码后带宽约为 7.34 GiB/s。这是链路编码上限，不是 SSD 必然能够持续达到的有效载荷吞吐；协议、控制器、队列和软件路径还会产生额外开销。

RAID 0 可能带来的提升

若三块 NVMe 组成 RAID 0，数据能够均匀条带化，并且三块盘、控制器与 PCIe 上行链路互不争用，则理论上限可从单盘 7.34 GiB/s 提升至约 22.02 GiB/s。实际值会受盘本身持续读能力、条带大小、RAID 实现、PCIe 拓扑、CPU/中断以及 XDS 多请求调度限制。

注意

RAID 0 提升的是源端并行带宽，不会自动解决非 0 卡读取、多卡调度、错误回传和稳定性问题，也不应在未测拓扑时直接按三倍性能承诺。

04 · BENCHMARK REVIEW

Benchmark 是否合理

总体设计合理，现有数据有明确参考价值。测试覆盖多种请求粒度和 io_depth，计时范围包含从 read_file() 提交到 drain_read() 完成的端到端 XDS 路径，并通过数据校验降低“只计时但没有正确传输”的风险。

合理之处

  • 覆盖 32K–512K 请求粒度及多个队列深度。
  • 区分独立命令与同进程连续 sweep。
  • 记录实际传输字节和端到端带宽。
  • 包含数据校验，并对 Python/C++ 实现做了对照。
  • 能够识别小请求固定开销和队列饱和趋势。

正式验收前应补齐

  • 每个参数至少重复 3–5 次，报告中位数、P95 和波动范围。
  • 统一扫描大小、文件布局、预热策略和运行方式。
  • 记录 SSD 温度、降频、PCIe 拓扑、CPU/IRQ 与系统日志。
  • 修复 I/O 完成错误回传，确保 PASS 代表严格成功。
  • 解决连续压力测试触发整机重启的问题。
评价

当前 benchmark 足以支持“单盘性能水平、参数趋势和异常点定位”等技术判断；如果用于产品指标承诺或版本正式验收，则需要补齐重复性、环境控制和故障闭环。

07 · CODE & STABILITY RISKS

基于 origin/main 的代码与稳定性风险

以下问题来自公开基线实现的源码分析。部分问题已在本地后续提交中出现修复尝试，但在进入正式支持范围前仍需由 OS/驱动团队评审并完成真机回归。

级别问题代码机制与影响建议验证
高非页对齐 NPU 地址处理get_pa_list() 成功时返回页内偏移，但调用方将任何非 0 返回值当作错误；随后用户态又主要按负值判断失败。可能出现请求未正常提交、返回语义却不明确的情况。覆盖 0、非 0 页内偏移及跨页地址，要求返回值和数据校验均正确。
高FIEMAP 首 extent 偏移计算基线按 file_offset % 4096 调整物理地址，没有使用 requested_offset - fe_logical。当 FIEMAP 返回的首 extent 逻辑起点与假设不同时，可能读到错误物理位置。测试非 0 文件偏移、extent 内偏移、4 KiB 边界和碎片文件。
高I/O 错误仍返回 0p2p_drain_read() 会累计 err_cnt，但函数末尾固定返回 0。上层可能把 NVMe 提交或完成失败误判为成功。注入提交失败与完成状态错误，要求 benchmark 明确 FAIL 并返回负 errno。
高batch 边界索引do_read_ios_batch() 在 idx++ 后直接访问 addr_off[idx]/align_size[idx]，缺少到达末尾后的边界判断，存在越界访问风险。覆盖单元素、正好结束、跨页、多元素及最后一段边界。
中FIEMAP 覆盖不足extent 容量按请求大小估算，且只检查映射总量是否大于请求，没有严格拒绝映射不足。碎片、稀疏或 unwritten 文件可能导致不完整映射。分页遍历 FIEMAP；拒绝洞、未写 extent 和覆盖不足；记录 extent 数。
中batch 连续性假设用户态按首请求的 offset 与 首请求 size × 数量 查询连续 FIEMAP，不能等价支持任意跳跃 offset 或不同 size 的通用批读。明确 API 约束，或逐请求构建映射；拒绝不满足连续性的输入。
中设备/namespace 假设请求中使用固定 nsid = 1，存储和文件系统路径也依赖“FIEMAP + 单 bdev”假设，多 namespace、CoW 或多设备文件系统未验证。建立设备矩阵，校验 namespace、bdev 与文件实际映射关系。
中非 0 卡与多卡接口携带设备编号，但当前环境非 0 卡读取未跑通，也没有多卡并发和拓扑回归。逐卡功能测试，再进行同根端口/跨根端口的并发吞吐和隔离测试。
中内核版本耦合NVMe 请求分配、完成回调及 blk_execute_rq_nowait() 等接口随内核版本变化，Linux 6.6 适配不能只以“编译通过”视为完成。由 OS 部门在 6.6.0 完成编译、加载、功能、错误注入和压力回归。

连续压测触发整机重启

高优先级

当前一次性运行完整 benchmark 压力矩阵可能直接触发整机重启，根因未知；把测试拆开、一次只运行一个参数组合时暂未出现问题。该现象是正式性能验收和长期运行前的阻塞项。

现有证据不足以将重启归因于 XDS 驱动、NVMe、NPU runtime、电源、散热或 watchdog 中的任何一项。定位时应保留上次启动的内核日志、pstore/ramoops、BMC SEL、电源与温度记录，并逐步增加 io_depth、并发卡数和持续时间，确定最小复现条件。

08 · CURRENT OPERATING ENVELOPE

当前建议的临时使用约束

在 Linux 6.6、多设备、多卡和稳定性问题完成闭环前，建议将公开版本的可用范围限制如下：

系统与设备
  • Linux 4.19.90；
  • 0 号 NPU 卡；
  • 单 NVMe、无 MD RAID；
  • 已验证的 CANN/驱动/PyTorch 组合。
文件与地址
  • 普通、非稀疏且已完全分配的文件；
  • 文件系统来源与指定 bdev 一致；
  • 偏移和长度至少按 4 KiB 对齐；
  • 优先使用顺序读和连续请求。
请求参数
  • 允许 256K/512K 上层请求；
  • 明确底层单命令最大 128 KiB；
  • 性能甜点优先从 256K/q256 起步；
  • 避免无收益地持续增加 io_depth。
运行与判定
  • 一次运行一个 benchmark 组合；
  • 保留数据校验，不只看带宽；
  • 每次检查 dmesg、设备错误和重启记录；
  • 出现错误或异常波动时不计为有效样本。
09 · RECOMMENDED ACTIONS

后续建议与优先级

1
先完成 Linux 6.6.0 和错误闭环由 OS/驱动部门基于内网经验评审公开版本适配，完成 NVMe/blk-mq API 兼容、I/O 错误回传、对齐和 batch 边界修复，并通过真机压力回归。
2
定位整机重启建立最小复现用例；采集 pstore/ramoops、BMC SEL、温度、功耗和内核日志；分别改变 io_depth、batch 间隔、持续时间和设备并发，区分软件缺陷与硬件保护。
3
建立设备与拓扑支持矩阵覆盖不同 NVMe、namespace、ext4/XFS、RAID 0、0/非 0 卡、单卡/多卡，以及同根端口和跨根端口拓扑。未验证组合不进入支持声明。
4
规范正式 benchmark统一 1 GiB 或更大的工作集、文件布局和预热方式；每点重复 3–5 次；报告中位数/P95、温度和错误状态；单点测试与连续稳定性测试分开执行。
5
优化小粒度软件路径以 requests/s 而非只以 GB/s 定位 32K–128K 开销，优先评估 C++ 原生提交、批量接口正确化、减少重复 open/FIEMAP/ioctl 和降低 drain 频率。
6
再评估多盘多卡性能功能与稳定性闭环后，测试 3 盘 RAID 0 与 8 卡并发，分别报告单卡、单盘、聚合吞吐及公平性，确认瓶颈位于 SSD、PCIe 上行、RAID、CPU 还是 XDS。
10 · FINAL CONCLUSION

最终结论

公开版 XDS 已证明单 NVMe 到 0 号 NPU HBM 的 P2P 读取路径可以在 Linux 4.19.90 上获得较好的大粒度顺序读性能，但尚未达到可对 Linux 6.6、多设备、多卡和高压力稳定性作正式支持承诺的状态。

性能

单盘峰值 6.31 GiB/s，达到 Gen4 x4 编码上限的 86%，大粒度性能较好；256K/q256 已接近峰值，是当前更均衡的参数点。

优化空间

32K–128K 仍受每请求软件开销限制，原生 C++ 可带来约 30%–50% 改善；继续盲目增加 io_depth 收益有限。

能力缺口

当前仅 0 号卡跑通。非 0 卡、多卡并发、RAID 0 和多种存储设备尚未形成完整验证，不能直接对标外部 3 盘 8 卡结果。

首要任务

先解决错误回传、对齐/batch 风险和整机重启，再由 OS/驱动部门完成 Linux 6.6.0 与设备矩阵验证，最后进入正式性能验收。

一句话判断

单盘大粒度性能已经可观；小粒度仍需优化；多盘多卡能力和高压力稳定性是当前真正的短板。
