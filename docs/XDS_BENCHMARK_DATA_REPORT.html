连续执行时序：异常集中在 128K q128

横轴按真实执行顺序排列：先跑完 32K 的各深度，再跑 64K，最后跑 128K。该图用于观察前序测试是否影响后续点。

第 16 个点，即 128K q128，从前一点 3.96 降至 3.50 GiB/s（−11.6%），但后续 q512/q1024 又升至 5.52/5.96。因此它更像单点波动、状态变化或测试噪声，不能据一次结果认定 q128 本身更慢。

相同参数的直接对比：连续运行并非整体变慢

只有三个参数点能够直接重合。128K q64/q128 的独立数据使用 800M；q1024 两边都是 1G，因此前两项仍存在扫描规模与临时文件物理布局差异。

核心判断：连续运行不是普遍性降速。q64 连续运行高 3.1%，q1024 高 1.0%；只有 q128 低 25.1%。所以现有数据更支持“连续矩阵中出现一个异常点”，而不是“连续运行导致所有性能下降”。

小请求表现为约 50K requests/s 的软件路径上限

对不超过 128K 的请求，带宽换算成 P2P 请求速率后，各粒度在高深度下都收敛到约 49K–53K requests/s。

这解释了为什么请求粒度近似翻倍时，带宽也接近翻倍：32K q1024 为 1.62 GiB/s、64K 为 3.17 GiB/s、128K 为 5.96 GiB/s，而每秒处理的 P2P 请求数变化不大。当前小请求更像受每请求的软件/驱动开销限制，而不是受 HBM 带宽限制。

C++ 与 Python 测试对比：小粒度的软件开销显著

C++ 数据来自 tools/single_npu_stream_bench.cpp。该脚本直接使用 ACL 和 file_p2p API，在原生循环中提交 read_file 并按 io_depth 调用 drain_read；相比 Python 测试，减少了 Python 解释器、对象构造及语言绑定等用户态开销。

小粒度下 C++ 快约 30%–50%。 请求越小，单位数据量需要的提交次数越多，Python 调度与绑定层的固定成本会被反复放大；改用原生 C++ 后，32K–128K 的 q128 带宽明显提高。
大粒度下收益很小。 256K q1024 时 C++ 约 6.02 GiB/s，与 Python 的 6.04 GiB/s 基本相同，说明此时性能已主要受 NVMe/PCIe 链路上限约束，继续降低语言层开销难以转化为带宽。

其中 Python 的 128K q128 是连续矩阵中的异常低点，因此该点的提升幅度不宜单独作为普遍结论；但 32K、64K 与 128K 的整体趋势一致，仍表明小请求场景存在较大的软件层固定开销。

因此不能用“独立最高 6.31 vs 连续最高 5.96”直接得出连续运行慢 5.5%。独立最高使用 512K，而连续数据只展示到 128K；两边没有测试同一个最高参数组合。

数据结论

链路结论
单盘最高 6.31 GiB/s，已接近 PCIe 4.0 x4 的实用上限。NPU x16 有大量余量。
调参结论
增大请求粒度比无限增加 io_depth 更有效。256K q256 已取得峰值的 94.8%。
连续性结论
连续运行整体趋势正常，没有系统性降速证据；128K q128 是应当复测的突出异常点。
实现开销结论
小粒度 q128 下 C++ 比 Python 快约 30%–50%，而 256K q1024 基本持平，说明软件固定开销主要影响小请求。
建议每点重复 ≥3 次 报告中位数/P95 统一使用 1G 文件 补测连续 256K/512K 重点复测 128K q128
